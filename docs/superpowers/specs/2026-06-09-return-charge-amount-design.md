# Дизайн: редактируемая сумма начисления при возврате

Дата: 2026-06-09
Статус: утверждён (брейншторм)

## Цель

При приёме возврата (например, вернули 7 единиц из 10) оператор должен сразу
видеть **посчитанную аренду за возвращаемые единицы** — по числу дней, что они
были на руках, — и иметь возможность **поправить эту сумму вручную** перед
подтверждением.

Сохранённая сумма — это **начисление за возврат**: она входит в общий итог
аренды (заменяет авторасчёт по этим единицам) и показывается в «Движениях».

## Принятые решения (из брейншторма)

1. **Сумма редактируется в окне возврата** (модалка «Принять возврат»), а не
   в таймлайне постфактум.
2. **Смысл суммы — начисление за возврат.** Сохраняется на движении возврата и
   входит в `base` (общий итог аренды) вместо авторасчёта по возвращённым
   единицам. Это не платёж (не `Payment`) и не справочное поле.
3. **Дни для авто-суммы — `days_avg`** (средние дни на единицу, уже считается и
   показывается в окне возврата). Для типового случая (всё выдано в один день)
   это точно; при выдаче партиями в разные дни — приблизительно, но оператор
   правит сумму руками. Точный пер-юнит FIFO для авто-подстановки не требуется.
4. **Штрафы за просрочку не трогаем.** Логика штрафа/залога/платежей в
   `compute_rental_billing` остаётся прежней — меняется только расчёт `base`.
5. **Обратная совместимость через `null`.** Старые движения, движения выдачи и
   списания (досрочное закрытие) имеют `amount = null` → для них биллинг
   считает по-старому (авторасчёт FIFO). Существующие тесты дают тот же результат.

## Архитектура

### Модель `Movement` (config/models.py)

Добавить одно поле:

```python
amount = models.DecimalField(
    _('Сумма начисления'),
    max_digits=12, decimal_places=2,
    null=True, blank=True,
    help_text=_('Аренда за возвращённые единицы. Только для возвратов; '
                'для выдачи и списаний — пусто.'),
)
```

- `null=True` — выдачи, списания и исторические возвраты остаются без суммы.
- Для новых возвратов поле всегда заполняется (явное значение оператора либо
  авторасчёт, если поле оставили пустым) — см. View.

Нужна миграция (`0017_movement_amount`), без data-миграции (старые строки = `null`).

### Биллинг (config/billing.py)

Новая функция расчёта базы по позиции — заменяет связку
`compute_item_unit_days() × price` внутри `compute_rental_billing`:

```python
def compute_item_base(item, as_of=None) -> Decimal:
    """Сумма аренды по позиции:
      * по каждому возврату — Movement.amount, либо авторасчёт (unit-days × price),
        если amount is None (старые/списания);
      * по ещё не возвращённым единицам — накопление unit-days × price до as_of.
    """
```

Реализация — тот же FIFO-проход, что в `compute_item_unit_days`:

- Каждая ВЫДАЧА сеет чанк `[qty, issue_dt]`.
- Каждый ВОЗВРАТ потребляет из старейших чанков; считает свои `unit_days`
  (для авторасчёта-fallback) и добавляет к базе:
  `movement.amount` если не `None`, иначе `unit_days × price`.
- Оставшиеся в очереди чанки (ещё на руках) добавляют `qty × дни(issue, as_of) × price`.

`compute_item_unit_days` **остаётся** (используется для `days_avg` в окне возврата
и в других местах — config/views.py:413, 939).

`compute_rental_billing`: строка `base += Decimal(days) * item.price_per_day`
заменяется на `base += compute_item_base(item, as_of=as_of)`. Штраф, залог,
платежи, `total_due` — без изменений.

Поскольку при `amount=None` `compute_item_base` даёт ровно тот же результат, что
текущий `unit_days × price`, существующие тесты биллинга остаются зелёными.

#### Хелпер для отображения суммы возврата

```python
def return_charge_map(rental, as_of=None) -> dict[int, Decimal]:
    """{movement_id: сумма} по каждому движению ВОЗВРАТА: amount, либо
    авторасчёт (unit_days × price) если amount is None. Для таймлайна."""
```

Тот же FIFO-проход; переиспользует логику `compute_item_base`.

### View (config/views.py)

**`_return_modal_context`** — для каждой строки уже считается `days_avg`,
`price_per_day`, `outstanding`. Добавить в контекст ничего нового по сути не
нужно (авто-сумма считается на клиенте из `days_avg × qty × price`). Поле ввода
суммы рендерится пустым; JS подставит значение при вводе количества.

**`RentalReturnView.post`** — в цикле сбора `plan`:

- Прочитать `amount_{pk}` (money-формат → `Decimal` через тот же strip-подход,
  что у прочих money-полей; см. `MoneyDecimalField` / config/forms.py).
- Валидация: если задано — `>= 0`; некорректное → ошибка в `errors` (модалка
  перерисовывается с введёнными значениями).
- Хранить введённое количество суммы рядом с `(item, qty)` в `plan`.

При создании `Movement` (возврат):

```python
amount = entered_amount
if amount is None:        # поле оставили пустым → берём авторасчёт за эту qty
    amount = compute_return_amount_for_qty(it, qty)  # FIFO дни этой qty × price
Movement.objects.create(..., kind=RETURN, qty=qty, amount=amount, note=note, ...)
```

`compute_return_amount_for_qty(item, qty)` — авторасчёт суммы за конкретное
возвращаемое количество (FIFO: потребить `qty` из старейших чанков на момент
`now`, сумма дней × price). Это «честный» дефолт, не зависящий от `days_avg`
(клиентская оценка нужна только для живого подсказывания в форме).

Списание (`RentalCloseView.post`) — `Movement.amount` не задаётся (`null`),
поведение биллинга для списаний прежнее.

### Окно возврата (config/templates/config/rentals/_return_modal.html)

Добавить колонку «Сумма» в таблицу. Итоговая структура строки:

| Товар | К возврату | Дн. | Цена/сут. | Возвращаю (qty) | **Сумма, сум (editable)** |

- Поле суммы: `<input name="amount_{{ row.item.pk }}" class="form-control form-control-sm text-end money-input" ...>`.
  Класс `money-input` даёт формат «210 000» и strip перед отправкой (money-input.js).
- Колонку «Σ за период» (текущий `row.billed` за полный outstanding) убираем или
  переименовываем — её роль теперь у редактируемого поля. Оставляем `Дн.` и
  `Цена/сут.` как ориентир.
- Футер «Итого» считает сумму по полям ввода вживую (JS).

### JS авто-подстановки (inline в модалке либо в money-input соседом)

- На `input` поля количества `qty_{pk}`: если соответствующее поле суммы **не
  тронуто руками** (`data-manual !== "1"`), подставить
  `amount = qty × days_avg × price` (значения — из `data-*` атрибутов строки),
  отформатировать через ту же логику, что money-input.
- На `input` поля суммы пользователем: пометить `data-manual = "1"` (больше не
  перезатирать авто-подстановкой).
- Пересчитывать футер «Итого» = сумма всех полей суммы.
- Привязка через делегирование/`htmx:afterSettle` (модалка приходит по htmx).

### Таймлайн (config/templates/config/rentals/_timeline.html)

Для движений возврата показать сумму из `return_charge_map` (по `m.id`):

```
Возврат · Корейская опалубка 1.5×1 · 7 — 210 000 сум
```

`_rental_card_context` (config/views.py:881) получает
`charges = billing.return_charge_map(rental)` и навешивает сумму прямо на объекты
движений (`m.charge = charges.get(m.id)`) перед сортировкой — шаблон тогда просто
проверяет `{% if m.charge %}` без отдельного фильтра. Для выдач/`None` сумма не
показывается.

## Тестирование

`tests/test_billing.py`:
- Возврат с явной `amount` → попадает в `base`, `total_due` соответствует.
- Возврат с переопределённой (отличной от авто) суммой → `base` берёт ручную.
- Возврат с `amount=None` → `base` == старый авторасчёт (совместимость).
- Частичный возврат (7 из 10): сумма за 7, оставшиеся 3 продолжают капать.

`tests/test_rental_flow.py`:
- POST возврата с `amount_{pk}` → `Movement.amount` сохранён.
- POST возврата с пустой суммой → сохранён авторасчёт за qty (не `null`).
- Невалидная сумма (отрицательная) → ошибка, движение не создано.
- Таймлайн отображает сумму возврата.

## Вне рамок (YAGNI)

- Штрафы за просрочку на конкретном возврате.
- Изменение общей модели штрафа/залога/платежей.
- Редактирование суммы возврата постфактум в таймлайне.
- Точный пер-юнит FIFO для **клиентской** авто-подсказки (серверный дефолт —
  точный; клиентский — оценка по `days_avg`, правится руками).
