# Дизайн: авто-подстановка срока возврата по выбранному товару

Дата: 2026-06-09
Статус: утверждён (брейншторм)

## Цель

На странице **«Новая аренда»** при выборе товара автоматически подставлять
**Срок и время возврата** (`Rental.due_date`), рассчитанный из срока аренды
товара, чтобы оператору не приходилось вводить дату вручную. Поле остаётся
**редактируемым** — ручное изменение приоритетнее авто-подстановки.

## Принятые решения (из брейншторма)

1. **Источник срока — существующее `Product.expected_max_days`** («Норма, дней
   (макс)»). Новое поле не заводим, миграция не нужна.
2. **При нескольких товарах берём самый короткий срок** (минимум
   `expected_max_days` среди выбранных товаров, у которых он задан).
3. **Ручное изменение «побеждает».** Как только оператор сам редактирует поле
   срока, авто-подстановка для этой формы отключается и значение не
   перезаписывается.
4. **Только страница создания аренды.** Правки существующей аренды
   (`RentalEditForm`, модалки) не трогаем — там срок уже задан.

## Правило расчёта

```
Срок возврата = Дата выдачи (created_at) + N дней
N = min( expected_max_days по всем выбранным товарам, у кого он задан )
```

- Время суток сохраняется: выдали в 14:30 → возврат в 14:30 через N дней
  (прибавляем `N × 24ч` к `created_at`; если `created_at` пуст — берём «сейчас»).
- Товары без заданной «Нормы» в расчёте **игнорируются**.
- Если ни у одного выбранного товара нет `expected_max_days` — поле **не
  трогаем** (оператор вводит вручную, как сегодня).
- `N ≥ 1` всегда, поэтому серверная валидация `due_date > created_at` и
  `due_date > now` выполняется автоматически.

## Когда срабатывает

- При выборе / смене / очистке / удалении товара — через htmx-события
  (`htmx:afterSettle`), т.к. пикер подменяет DOM.
- При изменении поля «Дата выдачи» (`created_at`) — пересчитать базу.
- Всё перечисленное — **только пока поле срока не тронуто вручную**.

## Архитектура

Изменение почти целиком клиентское (JS поверх существующего пикера). Серверная
часть — один data-атрибут в шаблоне.

### 1. Шаблон выбранного товара — `config/templates/config/rentals/_item_product_picked.html`

К скрытому `<input name="item_product">` добавить data-атрибут со сроком
(по аналогии с уже имеющимся `data-price`):

```html
<input type="hidden" name="{{ field_name|default:'item_product' }}"
       value="{{ product.pk }}"
       data-price="{{ product.daily_price|unlocalize }}"
       data-return-days="{{ product.expected_max_days|default:'' }}">
```

- Пусто (`''`), если у товара нет нормы — JS такие позиции пропускает.
- Шаблон переиспользуется в модалках (`field_name` отличается) — атрибут там
  безвреден.

### 2. Скрипт на странице создания — `config/templates/config/rentals/create.html`

В существующий `<script>` (блок `extra_js`) добавить:

- **Флаг `dueDateTouched`.** Инициализируется значением «поле уже непустое при
  загрузке» — чтобы повторная отрисовка формы после ошибки submit не затирала
  введённое оператором значение.
- **Слушатель ручного ввода** на `input[name="due_date"]` (`input`/`change`) →
  `dueDateTouched = true`. Программная установка `.value` событий не порождает,
  поэтому свои авто-обновления флаг не взводят.
- **Функция `syncDueDate()`:**
  1. Если `dueDateTouched` — выйти.
  2. Собрать `data-return-days` со всех `input[name="item_product"]`,
     оставить целые `≥ 1`.
  3. Если список пуст — выйти (поле не трогаем).
  4. `N = min(список)`.
  5. `base = ` значение `created_at` (через `new Date(...)`) или `new Date()`.
  6. `due = base + N × 86400000 мс`.
  7. Записать в `input[name="due_date"]` строкой формата
     `YYYY-MM-DDTHH:MM` (локальное время — собрать вручную из компонентов
     `getFullYear/Month/Date/Hours/Minutes` с нулевым паддингом; **не**
     `toISOString`, иначе уедет в UTC).
  8. Вызвать `recalc()` — пересчитать «Период / Σ к оплате».
- **Привязка `syncDueDate`:**
  - на `htmx:afterSettle` (рядом с существующими хэндлерами recalc);
  - на изменение `created_at` (в общем `input`/`change` слушателе или отдельно);
  - один вызов на `DOMContentLoaded` (на случай предвыбранных позиций при
    ре-рендере; сработает только если `dueDateTouched=false`).

Формат `datetime-local` совпадает с `_DT_LOCAL_FORMAT = '%Y-%m-%dT%H:%M'` в
`config/forms.py`.

## Без изменений

- Модель `Product` / `Rental` (поле `expected_max_days` уже есть).
- `RentalCreateForm`, валидация (`clean`) — работает как есть.
- View пикера (`ItemProductPickView` и пр.) — рендерят тот же шаблон.
- Правки существующей аренды и модалки.

## Тестирование

- **Серверный тест (pytest):** GET на `rental_item_product_pick` для товара с
  заданным `expected_max_days` рендерит `data-return-days="<N>"`; для товара без
  нормы — `data-return-days=""`. По аналогии с `tests/test_product_picker.py`.
- **JS-логика** покрывается ручной проверкой при приёмке (выбор товара
  заполняет срок; правка срока «замораживает» его; второй товар с меньшей
  нормой укорачивает срок, пока не тронуто вручную).
- Существующие тесты (`tests/`) не падают.

## Вне рамок (YAGNI)

- Новое отдельное поле «срок аренды» на товаре (используем `expected_max_days`).
- Авто-подстановка на странице правки существующей аренды.
- Кнопка «сбросить к авто-значению» после ручного редактирования.
- Серверный пересчёт `due_date` (логика остаётся клиентской подсказкой).
- Учёт разных сроков по позициям (одна общая дата на аренду, как сейчас).
