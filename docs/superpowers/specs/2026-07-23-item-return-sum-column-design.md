# Колонка «Сумма возврата» в таблице позиций аренды

**Дата:** 2026-07-23
**Статус:** одобрено, в реализации

## Цель
В таблице «Позиции» на карточке аренды добавить одну колонку **«Сумма
возврата»** — начисленную аренду по строке. Для каждой позиции показывает,
сколько составляет аренда за фактические дни (кол-во × дни × цена), а не только
ставку за сутки.

Семантика (выбрана заказчиком) — **полная аренда строки**: начисление за уже
возвращённые единицы **плюс** текущее начисление по единицам, ещё находящимся на
руках (до сегодня).

## Подход
Ничего нового считать не нужно — это ровно `billing.compute_item_base(item)`,
который уже существует и уже суммируется в итог аренды
(`compute_rental_billing` складывает `compute_item_base` по всем позициям).
Поэтому новая колонка — это **построчная расшифровка базы аренды**, и сумма
колонки по всем строкам сходится с `base` в блоке «Итог».

Значение считаем **во вью** и вешаем атрибутом на объект позиции — тот же
паттерн, что уже применён в `_rental_card_context` (`m.charge = charges.get(...)`,
`summary = ...`). Модельное `@property` отклонено: `billing` импортирует
`models`, обратный импорт создал бы цикл, а вся биллинг-логика по договорённости
живёт в `billing.py`.

Одна правка вью покрывает все места рендера таблицы, т.к. **все три** маршрута
идут через `_rental_card_context`:
- страница аренды (`RentalDetailView` → `detail.html` → `_card.html`);
- HTMX-фрагмент карточки в аккордеоне клиента (`rental_card` → `_card.html`);
- OOB-обновление после возврата/оплаты (`_oob_refresh.html`).

## Компоненты
- **`config/views.py` → `_rental_card_context`:**
  - посчитать `now = timezone.now()` один раз;
  - в цикле по `items`:
    `it.line_base = billing.compute_item_base(it, as_of=now).quantize(Decimal('0.01'))`;
  - передать тот же `now` в `compute_rental_billing(rental, as_of=now)` — чтобы
    построчные суммы точно сходились с итоговой базой (обе стороны на одну дату);
  - вернуть тот же `now` в ключе `'now'` контекста (убрать повторный вызов
    `timezone.now()`).
- **`config/templates/config/rentals/_items_table.html`:**
  - новый заголовок `<th class="text-end">{% trans "Сумма возврата" %}</th>`
    **сразу после «Возвращено»** (перед «Не возвращено»);
  - новая ячейка `<td class="text-end fw-semibold">{{ it.line_base }}</td>` в той
    же позиции. Формат — как у соседних денежных колонок (ru-локаль → «6000,00»,
    без суффикса валюты). Вставка в середину не затрагивает admin-блок кнопок в
    конце строки — структура для staff и admin остаётся согласованной.

## Крайние случаи
- **Позиция без движений** (ничего не выдавалось) → `compute_item_base` = 0.00 →
  «0,00».
- **Частично возвращённая позиция** → возвращённое (сохранённый `Movement.amount`
  или авто) + набежавшее по остатку на руках до сегодня.
- **Оператор правил сумму возврата вручную** → колонка использует сохранённый
  `Movement.amount`, поэтому совпадает с чеком возврата и итогом. Отдельной
  логики не требуется — она уже внутри `compute_item_base`.
- **Активная аренда** → значение «живое»: растёт по мере набега дней (как и
  общий `base`).
- Права/доступ не меняются: колонка видна всем, кто и так видит таблицу
  (staff/admin).

## Тесты
Добавить в `tests/test_rental_card.py` по образцу
`test_items_table_shows_line_daily_cost_column`:
- на позиции с выдачей **в прошлом** (чтобы база ≠ Σ/сут. и тест был
  показательным) отрендерить карточку и проверить, что:
  - в HTML присутствует заголовок «Сумма возврата»;
  - присутствует значение, равное `billing.compute_item_base(item)`
    (в ru-формате с запятой).
- (свойство согласованности) сумма `line_base` по позициям равна
  `summary['base']` — можно проверить прямым сравнением во вью-тесте.
- Полный набор `pytest` остаётся зелёным (существующие колонки и итог не
  меняются).

## Заметки
- Заголовок оставлен как «Сумма возврата» (термин заказчика). Т.к. семантика —
  полная аренда строки, для активных аренд это «сумма к оплате за строку»; при
  желании легко переименовать (напр. «Начислено», «Аренда, Σ») — правка одной
  строки шаблона.
