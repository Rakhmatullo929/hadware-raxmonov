# Дизайн: детализация аренд в карточке клиента (аккордеон)

Дата: 2026-07-09
Статус: утверждён (брейншторм)

## Цель

На **детальной странице клиента** оператор должен видеть **полную детализацию
каждой аренды прямо на месте**, не заходя в отдельную страницу аренды. По клику
на строку аренды под ней раскрывается её карточка со всей информацией — как на
странице аренды — и **с возможностью редактирования** (приём возврата, платежи,
позиции, закрытие, печать/PDF). Цель — чтобы оператору практически никогда не
приходилось открывать отдельную страницу аренды.

## Принятые решения (из брейншторма)

1. **Формат — аккордеон.** Клик по строке аренды в истории раскрывает детали
   прямо под строкой (не модалка, не отдельная колонка).
2. **Объём — полная детализация + редактирование.** Внутри раскрытой строки —
   те же блоки, что на странице аренды: денежная сводка (база/штраф/залог/к
   доплате), позиции (выдано/возвращено/не возвращено), движения, платежи, и все
   те же действия (возврат, платёж, +/редакт. позиции, закрытие, печать, PDF).
3. **Подход A — одна открытая аренда за раз, переиспользование карточки.**
   Редактирование в аренде завязано на **фиксированные ID** (`#modal-slot`,
   `#rental-summary`, `#rental-items`, `#rental-timeline`, `#rental-payments`) и
   OOB-обновление (`_oob_refresh.html`). Чтобы ID оставались уникальными,
   **в каждый момент раскрыта максимум одна аренда** — тогда весь механизм
   модалок/OOB работает **без изменений**. Мультиоткрытие с неймспейсом ID
   (вариант B) отклонено как избыточное и рискованное (трогает партиалы, общие
   со страницей аренды).
4. **Ленивая подгрузка.** Карточка аренды грузится HTMX-запросом только при
   раскрытии строки, поэтому страница клиента остаётся лёгкой даже при десятках
   аренд. Данные всегда свежие (перезапрос при каждом раскрытии).
5. **Компромисс v1 — свёрнутый заголовок может кратко устареть.** Ячейки
   свёрнутой строки-заголовка (статус, «на руках», суммы) считаются
   SQL-аннотацией и **не участвуют** в OOB-обновлении карточки. После правки
   внутри раскрытой карточки заголовок этой строки может показывать старые
   цифры до перезагрузки/переоткрытия (сама карточка при этом свежая).
   Принимаем это в v1. Строка-заголовок оформляется отдельным партиалом с `id`,
   чтобы позже добавить авто-обновление одним OOB-блоком (см. «Задел на апгрейд»).

## Архитектура

Новых полей/миграций БД **не требуется**. Изменения — рефактор шаблона аренды,
один новый лёгкий view-фрагмент, переработка истории аренд в карточке клиента и
маленький JS для поведения аккордеона.

### 1. Рефактор страницы аренды без изменения поведения

Тело [config/templates/config/rentals/detail.html](../../../config/templates/config/rentals/detail.html)
выносится в новый партиал `config/templates/config/rentals/_card.html` —
дословно те же обёртки с теми же ID:

```
#rental-summary   → _summary.html
#rental-items     → _items_table.html
row: col #rental-timeline (_timeline) + col #rental-payments (_payments)
#modal-slot
```

`detail.html` после рефактора:

```django
{% extends 'base.html' %}
{% load i18n %}
{% block title %}…{% endblock %}
{% block content %}{% include 'config/rentals/_card.html' %}{% endblock %}
```

Поведение страницы аренды не меняется — она рендерит тот же HTML. Прикрыто
существующими тестами (`tests/test_rental_flow.py` и пр.).

### 2. Новый view-фрагмент карточки аренды

**Роут** ([config/urls.py](../../../config/urls.py)):

```python
path('rentals/<int:pk>/card/', views.rental_card, name='rental_card'),
```

**View `rental_card(request, pk)`** — тот же доступ, что у страницы аренды
(`StaffOrAdminRequiredMixin`/аналог декоратора, как принято в проекте):

```python
@staff_or_admin_required
def rental_card(request, pk):
    rental = get_object_or_404(
        Rental.objects
        .select_related('customer', 'created_by', 'closed_by')
        .prefetch_related(
            'items__product',
            'items__movements__created_by',
            'payments',
        ),
        pk=pk,
    )
    ctx = _rental_card_context(rental)  # уже есть, config/views.py:1127
    return render(request, 'config/rentals/_card.html', ctx)
```

`is_admin`, `now`, `today` уже доступны: `is_admin` — из контекст-процессора
`navigation`, `now`/`today` — из `_rental_card_context`. Фрагмент не наследует
`base.html`, поэтому годится для инъекции в аккордеон.

> Реализация декоратора/примеси доступа — по образцу соседних rental-view; точное
> имя (`@staff_or_admin_required` vs `StaffOrAdminRequiredMixin`) выбирается на
> этапе реализации под существующий паттерн функциональных view в файле.

### 3. История аренд в карточке клиента → аккордеон

[config/templates/config/customers/detail.html](../../../config/templates/config/customers/detail.html):
блок «История аренд» переделывается так, чтобы каждая аренда состояла из двух
частей — **строка-заголовок** и **контейнер тела** под ней.

**Строка-заголовок** выносится в партиал
`config/templates/config/customers/_rental_history_row.html` с корневым
`id="crow-{{ r.id }}"` (задел на будущий OOB-апгрейд). Набор данных — тот же, что
сейчас в таблице: номер, «Создана», «Срок возврата», статус-бейдж, «Позиций»,
«На руках», «Оплачено/Залог», «Оформил». Отличия от текущей таблицы:

- Строка становится **кликабельным заголовком** (шеврон ▸/▾ слева, `role="button"`,
  `aria-expanded`, `aria-controls="rbody-{{ r.id }}"`, доступность с клавиатуры).
- Клик грузит карточку:
  `hx-get="{% url 'rental_card' r.id %}" hx-target="#rbody-{{ r.id }}"
  hx-swap="innerHTML"`.
- Кнопка **«Открыть»** (полная страница аренды, `rental_detail`) остаётся как
  второстепенное действие (на случай печати/шаринга ссылки).

Из-за вложенных блоков карточки (таблицы, `row`/`col`, модалка) детали **не
помещаются** в одну ячейку `<td>` семантически чисто. Поэтому история аренд
перестраивается из одной `<table>` в **список** (по аренде: `div.rental-acc-item`
с заголовком-строкой в виде fl‑раскладки внутри и контейнером тела под ним), либо
таблица с парами строк `<tr>` заголовок + `<tr><td colspan="N"><div id="rbody-…">`.
Решение (список vs colspan-строки) фиксируется на этапе плана; предпочтение —
**список карточек** (проще адаптив и вложенность). Свёрнутый заголовок при этом
сохраняет те же колонки визуально (выравнивание через утилиты сетки Bootstrap).

**Контейнер тела**: `<div id="rbody-{{ r.id }}" class="rental-acc-body"></div>` —
изначально пустой; сюда HTMX кладёт карточку.

### 4. Поведение «одна открытая за раз» (static/js)

Небольшой делегированный обработчик (новый `static/js/customer-rentals.js`,
подключается в карточке клиента или в `base.html` рядом с прочими):

- **Открытие**: перед загрузкой карточки в `#rbody-<id>` — очистить `innerHTML`
  у ранее открытого тела (убирает дубли ID и закрывает его модалку), снять с его
  заголовка признак раскрытия (`aria-expanded=false`, шеврон ▸), скрыть его.
- **Повторный клик по открытой** — свернуть (очистить тело), не грузить заново.
- **Индикация**: активному заголовку — `aria-expanded=true`, шеврон ▾, подсветка.

Так в DOM всегда максимум **один** комплект ID карточки → модалка/OOB работают
как на странице аренды. Реализация — на нативных событиях + `htmx` (например,
`htmx:beforeRequest`/делегированный `click`), без зависимости от Bootstrap
collapse (чтобы контролировать очистку тела и single-open явно).

### 5. Редактирование внутри аккордеона — без изменений

Все действия (возврат, платёж, +/редакт./удаление позиции, закрытие, изменение
аренды) остаются как есть: кнопки в `_summary`/`_items_table`/`_payments` делают
`hx-get … hx-target="#modal-slot"`, форма постит, view возвращает
[`_oob_refresh.html`](../../../config/templates/config/rentals/_oob_refresh.html),
OOB-своп находит `#rental-summary/#rental-items/#rental-timeline/#rental-payments`
**внутри раскрытой строки** и обновляет их на месте; пустое тело ответа очищает
`#modal-slot` → модалка закрывается. **Ни один view-действие и ни один общий
партиал не меняются.**

## Поток данных

```
[Страница клиента: клик по строке аренды #138]
        │ (JS: закрыть ранее открытую → очистить её #rbody)
        │ hx-get rentals/138/card/  → hx-target #rbody-138
        ▼
rental_card(138) → _rental_card_context → _card.html
        ▼
#rbody-138 наполняется карточкой (#rental-summary/#rental-items/
           #rental-timeline/#rental-payments + #modal-slot)
        │
[Оператор внутри карточки: «Принять возврат» и т.п.]
        │ hx-get … → #modal-slot (модалка)   → submit
        ▼
view-действие → _oob_refresh.html
        │ OOB-своп заменяет 4 блока внутри #rbody-138, очищает #modal-slot
        ▼
Карточка обновлена на месте. (Свёрнутый заголовок #crow-138 — см. компромисс v1.)
```

## Обработка ошибок и крайние случаи

- **Несуществующий/чужой `pk`** в `rental_card` → `Http404` (`get_object_or_404`).
- **Права** — тот же доступ, что у страницы аренды; аноним/без роли →
  редирект/403 (как у `RentalDetailView`). Админ-кнопки внутри карточки — под
  `{% if is_admin %}`, как сейчас.
- **Дубли ID при двойном быстром клике** — single-open JS очищает предыдущее
  тело **до** вставки нового; повторный клик по уже открытой — сворачивает.
- **OOB на странице аренды не ломается** — рефактор `detail.html` рендерит те же
  ID; `_oob_refresh.html` не меняется.
- **Пустая история** — блок показывает «Аренд пока нет.» (как сейчас).
- **Мобильный вид** — карточка использует `col-lg-6` для таймлайна/платежей →
  колонки стекаются; заголовок-строка сворачивается в компактный вид утилитами
  сетки.

## Тестирование

Тесты в `tests/` (запуск: `./venv/bin/python -m pytest`).

- **`rental_card` view**: GET авторизованным → 200; ответ содержит блоки
  `#rental-summary`, `#rental-items`, `#rental-timeline`, `#rental-payments`,
  `#modal-slot`; для несуществующего `pk` → 404; аноним/без роли →
  редирект/403 (как у `rental_detail`).
- **Регресс страницы аренды**: `rental_detail` после выноса `_card.html`
  по-прежнему 200 и содержит те же ключевые блоки/кнопки (существующие тесты
  `tests/test_rental_flow.py` должны остаться зелёными).
- **Редактирование через карточку-фрагмент**: POST возврата/платежа, вызванный
  в контексте фрагмента, возвращает `_oob_refresh.html` с четырьмя OOB-блоками
  (уже покрыто существующими тестами действий; при необходимости — точечная
  проверка, что фрагмент содержит цели OOB).
- **Карточка клиента**: `customer_detail` рендерит строки-заголовки с
  `hx-get` на `rental_card` и контейнерами `#rbody-<id>`; кнопка «Открыть»
  ведёт на `rental_detail`.

## Задел на апгрейд (вне рамок v1)

- **Авто-обновление свёрнутого заголовка** после правки: добавить в
  `_oob_refresh.html` OOB-блок `#crow-{{ rental.id }}` c
  `_rental_history_row.html`. htmx **безопасно игнорирует** OOB-цель, которой нет
  на странице аренды, поэтому изменение не затронет `rental_detail`. Потребует,
  чтобы view-действия отдавали в контекст аннотированные значения строки
  (`items_count`, `outstanding_total`, `paid_total`, `deposit_total`) — их считать
  дёшево для одной аренды. Оформляем строку партиалом с `id` уже в v1, чтобы
  апгрейд был точечным.
- Мультиоткрытие нескольких аренд одновременно (потребует неймспейс ID — вариант B).

## Вне рамок (YAGNI)

- Неймспейсинг ID и мультиоткрытие (вариант B).
- Мастер-деталь в две колонки (вариант C).
- Пагинация/ленивая подгрузка списка аренд в карточке клиента (список аренд
  одного клиента невелик; при необходимости — отдельная задача).
- Любые изменения в самих view-действиях (возврат/платёж/позиции/закрытие) и в
  общих партиалах аренды.
